# Database Architecture

Read this when choosing a database, designing a schema strategy, planning migrations, or defining data ownership. The data model outlives every other decision in the system — frameworks come and go; the schema and its data stay.

## Engine choice

**Default: one relational database (PostgreSQL unless the ecosystem strongly says otherwise).** Most product domains are relational — entities with relationships, integrity rules, and queries you haven't predicted yet. Postgres additionally covers the common "we might need X" cases without new infrastructure: JSONB for flexible fields, full-text search to a serious level, LISTEN/NOTIFY, row-level security (tenancy), and queue-like patterns (`SKIP LOCKED`) good enough for modest job volumes.

Reach for a second engine only when a *workload* demands it, and name the workload in the plan:

| Workload evidence | Engine class | Note |
|-------------------|--------------|------|
| Hot keys, sessions, rate counters, cache | Key-value (Redis) | Almost every system earns this one eventually; it's a cache, not a source of truth |
| Search relevance, facets, typo tolerance beyond Postgres FTS | Search (Elasticsearch/Meilisearch/Typesense) | A *projection* of the primary DB — rebuildable, tenant-filtered |
| High-volume append + time-window queries (metrics, telemetry) | Time-series / columnar | Don't torture B-trees with firehose appends |
| Truly document-shaped data, no cross-document queries/transactions | Document (MongoDB) | Rare as a *primary* for product data; JSONB covers most "flexible schema" needs |
| Multi-hop relationship traversal as the core query | Graph | Recommend only when joins demonstrably can't express it |

Every additional engine is a new backup story, failure mode, consistency boundary, and skill requirement — polyglot persistence must be earned per engine. Derived stores (cache, search index) must be rebuildable from the primary; say how in the plan.

## Schema strategy

- **Normalize first (3NF-ish); denormalize only with a named read-path justification** and a plan for keeping the copy consistent (trigger, transactional write, or projection). Duplicated data without an owner and sync story is a corruption generator.
- **Keys:** surrogate primary keys; UUIDv7 (time-ordered) when ids are exposed publicly, generated by distributed writers, or merged across environments; bigint identity when none of that applies and index compactness matters. Never expose sequential ids that enumerate customers (see IDOR / tenant enumeration in `security.md`).
- **Conventions to fix once, project-wide:** `created_at`/`updated_at` on every table (UTC, `timestamptz`); money as integer minor units or `numeric` — never floats; enums as lookup tables or check-constrained text (DB enums are annoying to migrate); soft delete (`deleted_at`) only where restore/audit is a real requirement — it complicates every query and unique constraint, so choose per table, not globally.
- **Constraints are architecture.** Foreign keys ON, NOT NULL by default, unique and check constraints for invariants the domain declares. "The app validates it" fails the day a second writer appears (job, migration script, support tooling). The DB is the last line of integrity — use it.
- **Avoid EAV / genericized "flexible" schemas** (`entities`, `attributes`, `values` tables). If fields genuinely vary per tenant/use, JSONB with validated shape beats EAV; real columns beat both.
- Large blobs (files, images) go to object storage; the DB stores metadata + keys.

For nontrivial domains, sketch the core entities (5–10) and their relationships as a Mermaid `erDiagram` in the plan (snippet in `diagrams.md`) — reviewers catch modeling errors in a diagram that they miss in prose.

## Indexing strategy

State the strategy, not just "add indexes":

- Index every foreign key (joins + FK-check performance) and every column pattern the hot queries filter/sort on — derive the initial index list from the Phase 3 request flows.
- Composite index column order: equality filters first, then range/sort. An index on `(tenant_id, created_at)` serves the pooled-SaaS "list my recent X" query shape; nearly every index in a pooled multi-tenant schema leads with `tenant_id`.
- Partial indexes for skewed predicates (`WHERE status = 'pending'` on a mostly-done table); expression indexes for computed lookups (`lower(email)`).
- Every index taxes writes and storage — don't pre-create speculative ones. The plan should name the *process*: slow-query log / `pg_stat_statements` review as a standing operational practice, indexes added from evidence.

## Migration strategy

- **Versioned, ordered migration files in the repo**, run by tooling (framework migrator, Flyway, dbmate…) — never hand-applied SQL to production. Migrations are code-reviewed like code.
- **Forward-only mindset:** write down-migrations where cheap, but design as if rollback means "roll forward with a fix" — data-destroying downs are a trap.
- **Zero-downtime discipline (expand → migrate → contract):** additive change first (new nullable column/table), backfill in batches, dual-write or sync period, switch reads, then contract (drop old) in a *later* release once nothing references it. Never rename in place; add-copy-drop. Beware long locks: adding NOT NULL columns with defaults, index builds without `CONCURRENTLY`, table rewrites on busy tables.
- Schema-per-tenant or DB-per-tenant multiplies every migration by N — the plan must include fleet migration orchestration (progressive rollout, per-tenant version tracking) or choose pooled tenancy partly to avoid it.

## Data ownership

Each module owns its tables exclusively; no other module reads or writes them directly. Cross-module access goes through the owning module's interface (in-process call), or via events/read-models where async fits. This single rule is what keeps a modular monolith modular and what makes later service extraction possible — shared tables are the coupling that hurts most and shows last. In the plan's Component Responsibilities section, list each module *with the data it owns*; a table nobody owns or two modules both write is a design error to fix on paper, where it's cheap.

Reporting/analytics reads that cross all modules go to a replica or warehouse, not to the primary through the back door.

## Scaling and operations

Scale the read path in cost order — each step is an order of magnitude cheaper than the next: query/index tuning → caching (with explicit invalidation story) → read replicas (accepting replication-lag semantics — read-your-own-writes needs care) → partitioning large tables (time-series, per-tenant pruning) → sharding, which is an application rewrite; if projected scale genuinely implies sharding, note the future shard key *now* (almost always `tenant_id`) and keep it on every table. Write-path scaling: batch, queue spikes, move append-heavy workloads to fitting stores.

Operational requirements that belong in every plan's Deployment section: automated backups **with restore actually tested**, point-in-time recovery for anything transactional, stated RPO/RTO, connection pooling (PgBouncer or equivalent — the classic serverless/high-concurrency failure), and data retention/deletion policy (compliance-driven; deletion must reach backups' expiry story too).
